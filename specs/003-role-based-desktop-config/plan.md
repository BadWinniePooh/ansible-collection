# Implementation Plan: Role-Based Desktop Configuration

**Branch**: `003-role-based-desktop-config` | **Date**: 2026-05-05 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `specs/003-role-based-desktop-config/spec.md`

## Summary

Refactor the monolithic `playbooks/setup-desktop.yml` into nine discrete Ansible roles
(`base_packages`, `dotnet`, `firefox`, `homebrew`, `github_cli`, `fnm`, `claude_code`,
`remote_desktop`, `vscode`). Replace `configurations/configure-linux.yml` with a single-play
orchestrator that loads a provider+platform vars file (`configurations/vars/hetzner-linux.yml`)
and executes the declared `active_roles` list via `include_role` in a loop.

## Technical Context

**Language/Version**: YAML (Ansible playbooks + roles), Jinja2 templates — ansible-core as installed via `pipx` on the control node
**Primary Dependencies**: `ansible-core` (pipx venv), `community.general` collection (snap module), `hetzner.hcloud` collection
**Storage**: N/A — no database or persistent state
**Testing**: `ansible-lint` (zero errors on modified files), manual test run against Hetzner Linux host
**Target Platform**: Ubuntu/Debian Linux (Hetzner Cloud CPX/CX server)
**Project Type**: Infrastructure automation collection (roles + playbooks)
**Performance Goals**: N/A
**Constraints**: All tasks idempotent; no plaintext secrets; shell tasks require `creates` or `changed_when` guards
**Scale/Scope**: 9 new roles (~25 task files), 2 new config files, 1 rewritten entry-point

## Constitution Check

*GATE: Must pass before Phase 0 research. Re-check after Phase 1 design.*

- [x] **I. Provider Abstraction** — Provider-specific variable *data* (`configurations/vars/hetzner-linux.yml`) is scoped by provider+platform naming convention, mirroring the existing `provisioners/hetzner-linux-up.yml` pattern. The constitution's restriction on "provider-specific logic" applies to provisioning/teardown operations (cloud API calls), not to configuration vars files. No cross-provider leakage into shared roles. **PASS**
- [x] **II. Idempotent Playbooks** — All tasks migrated 1:1 from `setup-desktop.yml` which is already idempotent. Shell tasks carry `creates` guards (homebrew installer, brew installs, claude code install) or `changed_when` (fnm default). Apt/service/lineinfile/file tasks use declarative state. **PASS**
- [x] **III. Vault-Only Secrets** — No secrets introduced by this feature. Existing vault-pattern vars (`desktop_users`, `my_hetzner_config`) flow through unchanged. **PASS**
- [ ] **IV. Docker-First Local Testing** — No Docker provider path included. `configurations/vars/docker-linux.yml` is not created. Feature spec explicitly excludes Docker support. See Complexity Tracking for justification. **VIOLATION — JUSTIFIED**
- [x] **V. Declarative Configuration** — All state in YAML tasks/roles/group_vars. Shell tasks exist only for tools without Ansible modules (Linuxbrew installer, `brew` CLI, `fnm` CLI, `npm`); these are inherited from the original playbook. **PASS**

## Implementation Order

Tasks are ordered to minimize risk and allow incremental verification:

1. **Create `configurations/vars/hetzner-linux.yml`** — declares `active_roles` list (no code risk)
2. **Create `roles/base_packages/`** — foundational packages; simplest role to verify
3. **Create `roles/dotnet/`** — simple apt install
4. **Create `roles/firefox/`** — simple apt install
5. **Create `roles/homebrew/`** — most complex single role; installs Linuxbrew + per-user config
6. **Create `roles/github_cli/`** — depends on homebrew; brew install
7. **Create `roles/fnm/`** — depends on homebrew; multi-step per-user config
8. **Create `roles/claude_code/`** — depends on fnm; per-user npm install
9. **Create `roles/remote_desktop/`** — largest role; tag all tasks
10. **Create `roles/vscode/`** — snap install; simple
11. **Rewrite `configurations/configure-linux.yml`** — wire everything together after all roles exist
12. **Verify**: `ansible-lint` on all modified files; manual run against Hetzner test host

## Key Design Decisions

### configure-linux.yml structure

```yaml
---
- import_playbook: ../playbooks/setup-users.yml

- name: Configure Linux desktop environment
  hosts: all
  become: true

  vars:
    desktop_user_names: "{{ desktop_users | map(attribute='name') | list }}"

  vars_files:
    - "vars/{{ provider }}-{{ platform }}.yml"

  tasks:
    - name: Run desktop roles
      ansible.builtin.include_role:
        name: "{{ item }}"
      loop: "{{ active_roles }}"
```

### Handler naming

`Cleanup apt cache` and `Restart XRDP` are defined in each role that needs them. Within a single
play run, Ansible deduplicates handlers by name — the first definition registered wins. Since the
handlers are functionally identical across roles, this is safe.

### Apt cache update placement

The apt cache update task (originally a `pre_tasks` item in `setup-desktop.yml`) moves into
`base_packages/tasks/main.yml` as the first task. Since `base_packages` is always first in
`active_roles`, this preserves the execution order.

### `become: false` tasks

Tasks that must NOT run as root:
- `homebrew`: installer script and `brew shellenv` PATH lines — lineinfile runs as each user
- `github_cli`: `brew install gh`
- `fnm`: `brew install fnm`; `fnm install 24` and `fnm default 24` use `become_user: "{{ item }}"`
- `claude_code`: `npm install -g` uses `become_user: "{{ item }}"`

All other tasks inherit `become: true` from the play level.
